# DEMO_SCENARIO.md — сквозной сценарий MVP

## Цель демо
За 2 минуты показать, что система превращает корпус документов в объяснимый портфель
проверяемых R&D-гипотез, а не просто пересказывает литературу.

## Домен MVP
Алюминиевые сплавы, термообработка/старение, прочность/твёрдость, пластичность и стоимость.

Почему этот домен выбран:
- в открытой литературе есть понятные связи `режим обработки -> микроструктура -> свойство`;
- легко показать KPI и ограничения: повысить прочность/твёрдость, не потерять пластичность,
  не выйти за бюджет;
- данные можно представить как статьи + внутренний CSV исторических экспериментов.

## Источники для тестового корпуса
MVP использует подготовленные текстовые выжимки в `sample_docs/`, чтобы не зависеть от качества
PDF-парсинга во время демо.

Рекомендуемые реальные источники для дальнейшего расширения:
- Open-access статья: https://link.springer.com/article/10.1007/s10853-020-05740-x
- Обоснование NLP-подхода к материаловедческой литературе:
  https://www.nature.com/articles/s41586-019-1335-8
- Пример domain-specific NLP для материаловедения:
  https://www.nature.com/articles/s41524-022-00784-w
- Future extension для структурированных данных:
  https://docs.materialsproject.org/

## Вход пользователя
Пользователь выбирает папку корпуса и вводит KPI-контракт:

```json
{
  "target": {
    "metric": "strength",
    "direction": "increase",
    "minimum_delta_percent": 10
  },
  "constraints": [
    { "metric": "cost", "op": "<=", "value": 5, "unit": "percent" },
    { "metric": "ductility_loss", "op": "<=", "value": 3, "unit": "percent" }
  ]
}
```

## Ожидаемый demo flow
1. Пользователь выбирает `sample_docs/`.
2. Frontend отправляет KPI-контракт в API.
3. Python Agent возвращает `claims`, `entities`, `edges` из `fixtures/extract_response.json`
   или из простого parser/LLM extraction.
4. Rust Engine строит граф и ранжирует гипотезы.
5. Frontend показывает board из 3-5 гипотез.
6. Пользователь открывает карточку гипотезы и видит:
   - краткую формулировку;
   - trace по claims и источникам;
   - score breakdown;
   - риски и недостающие доказательства;
   - DOE-план для проверки.
7. Пользователь делает rerun: например, исключает дорогой фактор или повышает вес `cost`.
8. Board пересобирается без повторного извлечения документов.

## Успешное демо
Демо считается успешным, если зритель может ответить на четыре вопроса по каждой top-гипотезе:
- Почему система предложила именно это?
- На каких источниках это основано?
- Что может пойти не так?
- Как это проверить экспериментом?

# Фабрика гипотез — описание решения

## Что это

Инструмент для автоматической генерации и приоритизации проверяемых гипотез по снижению
потерь металлов с хвостами обогатительных фабрик. На вход — отчёт института по хвостам
(xlsx), корпус литературы (учебники кейса + открытые источники) и целевой KPI.
На выходе — диагноз потерь в тоннах и деньгах + ранжированный портфель гипотез
с обоснованием, экономикой и планом эксперимента.

Это не чат с LLM. Диагноз ставит детерминированный парсер, гипотезы находит
детерминированный алгоритм на графе знаний. LLM используется только для чтения
литературы и (P1) объяснения результатов.

---

## Как работает

### 1. Вход
- **KPI-контракт** — цель (снизить извлекаемые потери element_28) + ограничения
  (capex-класс) + цены металлов (данные кейса анонимизированы — цену вводит пользователь)
- **Отчёт по хвостам** — реальный xlsx кейса: потери по классам крупности × минералогии
- **Корпус** — учебники кейса (PDF) + открытые статьи (open access, с URL и лицензией)
- **Domain Pack** — YAML со всей доменной семантикой (`packs/flotation-v1.yaml`)
- **Конфиг фабрики** — доступное оборудование (`factories/<id>.yaml`)

### 2. Диагноз потерь (Python sidecar, детерминированный — БЕЗ LLM)
- Якорный парсер xlsx: устойчив к плавающей структуре, битым ячейкам (#REF!),
  секциям породных/пирротиновых хвостов — проверен на всех 4 фабриках кейса
- Каждая ячейка потерь: тонны, извлекаемость, **диагноз причины** по правилам из pack
  (закрытые сростки в крупных классах → недоизмельчение; раскрытый минерал в −10 мкм →
  переизмельчение; раскрытый в средних классах → кинетика флотации; силикаты → неизвлекаемо)
- Каждая цифра несёт `cell_ref` — ссылку на ячейку исходного отчёта
- Проблемы данных не скрываются — попадают в панель Data Readiness

### 3. Извлечение знаний (Python sidecar) — RAG: retrieval
- PDF-корпус → чанки с номерами страниц → эмбеддинги, вектор-индекс
- LLM извлекает структурированные **claims**: `рычаг → механизм → эффект`,
  с полярностью и постраничной привязкой к источнику, под строгим валидатором
- Нормализация терминологии (Entity Resolver: гидроциклон/циклон/ГЦ → одна сущность)

### 4. Граф знаний (Rust core)
- Из claims строится граф: `рычаг → механизм → диагноз → KPI`; диагноз-узлы получают
  тоннаж из диагностики, рычаги — признак доступности из конфига фабрики
- Граф фиксируется как **snapshot с хешем** — граница воспроизводимости

### 5. Discovery Engine (Rust, детерминированный)

| Оператор | Что ищет | Гипотеза | Статус |
|---|---|---|---|
| `mechanism_path` | путь от KPI через диагноз с тоннажом до доступного рычага | «замени насадки ГЦ 12→8» | P0 |
| `substitution` | альтернативный рычаг с тем же механизмом | «грохот вместо гидроциклона» | P0 |
| `gap` | диагноз с тоннажом без доступного рычага | «нужна магнитная сепарация» (новое оборудование) | P0 |
| `contradiction` | claims с противоположным эффектом | «найди граничное условие» | P1 |
| `analogy_transfer` | механизм из смежного передела без переноса | «примени приём из Cu-цикла» | P1 |

Операторы не знают о флотации — работают с типами рёбер и тегами. Доменная начинка — из pack.

### 6. Оценка: деньги, а не абстрактные баллы
Каждая гипотеза получает **economic_effect**: адресуемые тонны (из диагноза) ×
консервативный диапазон прироста извлечения (из литературы, с trace) × цена металла =
**$ в год, диапазоном, с раскрытыми допущениями**. Скоринг по 6 осям
(`kpi_impact`, `evidence`, `plausibility`, `cost`, `risk`, `novelty`), хард-фильтры
по capex и доступности оборудования. Skeptic-rules ищут слабые места.

### 7. Benchmark против экспертов — встроенная валидация
В данных кейса есть гипотезы реальных мозговых штурмов по всем 4 фабрикам (27 штук) —
мы используем их как ground truth: система показывает, сколько экспертных гипотез
воспроизведено и какие новые найдены. Плюс live-прогон на фабрике, которую система
«не видела» (смена одного YAML).

### 8. Вывод
Для каждой гипотезы: обоснование → **цепочка происхождения, кликабельная до ячейки
исходного xlsx и страницы учебника** → экономика с допущениями → скептический разбор →
план эксперимента (DOE). Экспорт портфеля в CSV/JSON.

## Управление портфелем

Структурированные действия (`exclude_factor`, `change_weight`, `add_constraint`,
`relax_constraint`, `change_price`): исключить рычаг, поменять цену металла — портфель
пересчитывается за секунды с подсветкой изменений, snapshot-hash не меняется.

**P1 (в разработке):** LLM-нарратив гипотез, novelty-оценка против открытого корпуса,
Budget What-If и Manager View.

## Стек

- **Rust** — детерминированный движок, граф (petgraph), API (axum)
- **Python + FastAPI** — диагност xlsx (детерминированный), LLM-извлечение, эмбеддинги
- **TypeScript/React** — heatmap диагноза, портфель, карточка гипотезы, benchmark view
- **LLM** — только на этапах extract (P0) и narrate (P1)

---

## Ключевое отличие от «чата с ИИ»

Воспроизводимость: те же данные + KPI + pack → байт-в-байт тот же результат (hash).
Объяснимость: каждое число прослеживается до ячейки отчёта или страницы источника.
LLM не принимает решений: диагноз, гипотезы, ранжирование, деньги — детерминированный код.
Универсальность: домен целиком живёт в YAML-pack — смена передела не трогает движок.

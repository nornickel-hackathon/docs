## Как нас будут проверять (слова эксперта, без интерпретаций)

1. **Три текстовых промта**: два — по обогащению, один — по металлургии
   («мы будем говорить про получаемые шлаки... там не будет хвостов в текущем
   понимании и данных по Excel. Там будут данные в формате схемы цепи аппаратов,
   их описание»).
2. **Скрытая фабрика**: «У нас есть ещё один отчёт по фабрике, который мы вам не
   давали. По ней будет один из вопросов... Мы дадим на вход данные по оборудованию,
   которое на ней используется. Отчёт аналогичный тому, что есть в Excel».
3. **Ограничения — текстом**: «Все ограничения будут задаваться текстом. Мы не
   ожидаем каких-то выпадающих списков». Формат — верхнеуровневый: время, деньги,
   объёмы; упоминались и ограничения по сбыту металла.
4. **Запуск в их среде**: «Запускать решение будем в своей среде. Соответственно,
   нужен докер и docker compose». LLM из их контура — только Яндекс (Claude/GPT
   проверяли — недоступны). Доступ в интернет для открытых источников — будет.
5. Оценка — экспертная, вручную: «будем их сдавать, смотреть, что получится, и
   экспертно давать оценку». На чистоту кода смотреть не будут, «достаточно baseline».
6. Сдача: Яндекс.Диск приоритетнее гита. Видео смотрят только у финалистов.

## Подтверждено — НЕ трогать и продавать смелее

- **Детерминизм + trace = ядро оценки.** Эксперт: «недетерминированный pipeline —
  это и есть генерация знаний LLM», то есть чёрный ящик, которого просили избегать.
  Наш детерминированный engine с trace до ячейки — прямой ответ на критерий.
- **Wow-формат гипотезы — это наша карточка.** Эксперт перечислил 5 элементов
  гипотезы, вызывающей «вау-эффект»: (1) проблема, выявленная из входных данных →
  (2) почему она возникает → (3) как её принято решать в мировой практике →
  (4) что рекомендуется сделать здесь и сейчас → (5) почему эта проблема важнее
  других (наибольший потенциал + подтверждение внедрением на других активах).
  У нас: trace до ячейки xlsx → механизм → claims из литературы → рычаг →
  addressable tons/$ и ранжирование. Совпадение почти построчное — в демо и
  презентации проговорить эти 5 битов в этом порядке.
- **Data Readiness — заявленный плюс**: «если система отнесёт это к проблемам
  качества исходных данных, это будет плюсом». У нас готово (КГМК: 33 ref_error, 62.5%).
- **Экономика «приветствуется, но не ожидается»** — наш economic_effect и roadmap
  перевыполняют ожидания. Достаточный минимум по словам эксперта: механизм,
  новизна, риск, ценность — все четыре оси в скоринге есть.
- **Открытые источники разрешены и поощряются**, сеть будет.
- **Идеальный кейс пользователя** в пересказе эксперта (KPI + ограничения по сбыту →
  гипотезы с причинами → порядок тестирования при внедрении) = наш
  KpiContract + портфель + roadmap, включая rerun по цене металла.
- Подтверждена и диагностическая логика: по словам эксперта, зависимость доли
  извлекаемого металла от класса крупности — «там даже регрессия, достаточно
  высокий R²».

## Зазоры → задачи по ролям

### P0 · Роль 1 (Rust) + Роль 4 — Docker-стек всего решения

«Запускать будем в своей среде, нужен докер и docker compose» — это требование
запуска, наравне с видео. Сейчас: у backend нет Dockerfile, единого compose нет
(frontend/docker-compose.yml — только фронт; agent-system/compose.yaml — сайдкар+PG).

1. `backend/Dockerfile` — multi-stage: `cargo build --release -p platform` →
   slim-runtime; env `NORNIKEL_ROOT`, `SIDECAR_URL`.
2. Корневой `compose.yaml` уровня проекта: `platform` (8080, NORNIKEL_ROOT=/data/docs,
   SIDECAR_URL=http://sidecar:8765) + `sidecar` (uvicorn; после фиксов стартует без
   .env — не делать его обязательным) + `frontend` (nginx: `/api` proxy на
   backend:8080 уже настроен в nginx.conf) + volume: `docs/` и `norn-hack/` read-only.
   Postgres — опциональный профиль (RAG), демо без него.
3. README уровня проекта: «`docker compose up` → localhost:80, демо-скрипт см.
   DEMO_SCENARIO.md».

**Приёмка:** на чистой машине `docker compose up --build` → полный скрипт из
`DEMO_SCENARIO.md` проходит (run/benchmark/trace/roadmap/rerun/factories/export).

### P0 · Роль 1 (Rust) — onboarding скрытой фабрики

Проверено по коду: `/run` для неизвестной фабрики падает в 3 местах —
`infrastructure/file_factory_repository.rs:22-26` (нет `factories/<id>.yaml` → Err),
`infrastructure/file_diagnostics_source.rs:22-27` (только фикстура),
`infrastructure/http_sidecar.rs:62-70` (даже с живым сайдкаром сначала обязательно
грузится файловая фикстура — из неё берётся `file_path`). Сайдкар при этом к новым
xlsx готов: `DiagnoseRequest.file_path` — параметр, якорный парсер проверен на
4 разных структурах файлов.

1. Дельта контракта (сначала `CONTRACTS.md`): `RunRequest.source_file: Option<String>` —
   путь к xlsx новой фабрики.
2. Если `source_file` задан — `/diagnose` дергается напрямую с этим путём (минуя
   обязательный fallback-прегруз в `http_sidecar.rs`); файловый fallback остаётся
   только для известных фабрик.
3. `factories.load`: нет yaml → `FactoryConfig` «всё оборудование доступно» +
   пометка в data_readiness/board: `equipment list not provided — hard-filter disabled`.
   (Список оборудования, который дадут организаторы, Роль 4 руками кладёт в
   `factories/<id>.yaml` за минуты — тогда hard-filter включается.)
4. Файлы: `web/dto.rs`, `application/run.rs`, `infrastructure/{file_factory_repository,
   file_diagnostics_source, http_sidecar}.rs`.

**Приёмка (репетиция с Ролью 4):** временно спрятать `fixtures/diagnostics_nof_med.json`
и `factories/nof_med.yaml` → `POST /run {factory_id:"nof_med", source_file:"norn-hack/Пример 3/Хвосты НОФ мед.xlsx"}`
с живым сайдкаром выдаёт board с гипотезами и пометкой про оборудование, ничего не падает.

### P1 · Роль 2 (Python) — «текст → ограничения»

Проверять будут промтами, у нас — кнопки. Это НЕ возврат NLP-чата: нужен транслятор
текста в **уже существующие** rerun-действия и поля контракта.

1. `POST /parse_constraints` `{text, kpi_contract?}` →
   `{actions: RerunAction[], kpi_contract_patch?, unparsed: string[]}`.
   Реализация — Yandex structured output по образцу `app/pipeline/extract/service.py`
   (`client.beta.chat.completions.parse`, StrictModel-схема ответа); клиент уже есть
   в `app/infra/llm.py`.
2. Mock-режим на правилах (демо-страховка без LLM): regex-паттерны для «цена
   элемента 28/29 = N», «бюджет/без капзатрат/только настройки» (→ max_capex),
   «исключи X» (→ exclude_factor через synonyms из pack). При
   `SIDECAR_LLM_ENABLED=false` работает только rules-путь.
3. Всё, что не распозналось, — честно в `unparsed` (фронт покажет), не выдумывать.

**Приёмка:** «цена никеля выросла вдвое, капзатраты запрещены, скандий исключи» →
`change_price ×2 element_28` + `max_capex=1` + `exclude_factor`; кривой текст →
пустые actions + весь текст в unparsed, не 500.

### P1 · Роль 3 (Frontend) — текстовое поле ограничений

1. Текстовое поле рядом с RerunControls («задай ограничение словами») →
   `POST /api/constraints/parse` → полученные actions применить существующим
   механизмом rerun (`applyRerun`/`POST /rerun`), показать бейджи применённых
   действий + `unparsed` серым («не понял: ...»).
2. Кнопки остаются как fallback — ничего не выпиливать.
3. Зависимость от Роли 1: тонкий прокси-роут `/constraints/parse` → сайдкар
   (фронт говорит только с Rust, 8080). Зафиксировать в CONTRACTS.md вместе с Ролью 1.

**Приёмка:** в UI ввод «цена элемента 28 = 33000» меняет портфель так же, как
кнопка change_price; при недоступном сайдкаре поле показывает понятную ошибку,
кнопки работают.

### P2 · Роль 1 (Rust) — claims-only run (металлургический промт)

Один из трёх промтов — про шлаки, схемы цепей аппаратов, **без xlsx**. Полную
металлургию не делаем; делаем грациозную деградацию + слайд.

1. `/run` без диагностики (нет ни `source_file`, ни фикстуры) → не падать:
   пустой `DiagnosticsReport`, гипотезы из claims-графа без `addressable_tons`
   и `economic_effect` (kpi_impact только из claims), пометка
   `no quantitative diagnostics — hypotheses from literature graph only`.
2. Роль 4: слайд future work «домен = pack: metallurgy-v1 — это новый yaml +
   /extract по корпусу о шлаках, ядро не меняется» + честный ответ на промт
   металлургии через claims-only режим.

**Приёмка:** `POST /run {factory_id:"unknown_slag"}` возвращает board с
literature-гипотезами и пометкой, а не 500.

### P0/P1 · Роль 4 (QA/Data/Demo)

1. **P0 Питч новизны — переформулировать** (код не трогать). У жюри: «новизна — это
   то, что мы не дали на вход... всё, что прямо сейчас не используется в компании,

4. Сдача: **Яндекс.Диск приоритетнее гита**; видео смотрят только у финалистов

   потому что запускать будут сами).
5. В демо-сценарий добавить бит про 5 элементов wow-гипотезы (проговорить их
   порядком карточки) и бит про ограничение текстом.
